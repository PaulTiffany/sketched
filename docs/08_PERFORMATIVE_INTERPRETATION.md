# 08 · Performative Interpretation

A simple reading of what the parts _mean_. Clear enough for builders, honest
enough for the philosophically inclined. You do not need this page to use
Sketched — but it explains why the pieces are shaped the way they are.

| Part | Performative reading |
| --- | --- |
| **The stage** | The shared medium. A bounded surface everyone can witness. |
| **A layer** | A performed claim over time — an assertion placed on the stage. |
| **A knob** | A bounded parameter gesture. You act within a range, not without limit. |
| **A generated scene** | A provisional hypothesis. It is offered, not asserted as real. |
| **The human/video layer** | Embodied presence. Not a claim among claims — the ground the claims are made near. |
| **A mask** | A negotiated boundary between presence and generation. |
| **Shake** | Revocation / forgetting. The authority to un-say. |
| **The audit log** | Memory under constraint. What was said, by whom, and whether it was allowed. |
| **Consent** | The act that turns a _possible_ mutation into an _authorized_ one. |

## Why it lands as code

- Claims are provisional, so generated layers are **shakeable** and marked as
  hypotheses, never as truth (see [`09`](09_SAFETY_CONSENT_BOUNDARIES.md)).
- Presence is the ground, so the human layer is **preserved and un-agent-ownable**.
- Gestures are bounded, so **knobs** have ranges and write **keyframes**.
- Un-saying is real, so **shake** is a first-class, scoped operation.
- Memory is constrained, so the **audit log is append-only** — you cannot rewrite
  what was witnessed.
- Authorization is an act, so **consent** is a discrete, recorded decision, not an
  ambient assumption.

The short version: Sketched makes the governance of a performance into the actual
mechanics of the toy. That is the whole trick.
